# T2. 结果确定性优于过程确定性

## 1. 核心公理

先定义什么才算`正确`并验证它；不要试图通过事无巨细地管控步骤来让 AI 变得可靠。

## 2. 深度推演

### 2.1 过程确定性的上限

传统编程的信心来自过程确定性：每一行代码都在你的控制下，每个分支、每个边界情况都被你设计过。这种确定性是有形的，也是我们多年来训练出的核心能力——把结果转化为程序行为。但这个模式在 AI 系统中会遇到根本性的困境。

当你试图通过规则来约束 AI 的行为时，你会发现自己陷入一个无穷的防御性编程循环。一个规则修复了一个问题，却在其他情况下引入了新问题。你添加更多规则来处理边界情况，但边界情况是无限的。最终，你会发现自己维护的规则比产品逻辑还多。这正是我在 某 AI 翻译项目中遇到的困境：分块、重试、术语表、断点续跑、中文字符检测、超时处理——每一个都是为了防止某个特定的失败模式。但这些防御性规则本身就成了系统的主要复杂度来源，而且永远无法覆盖所有情况。

过程确定性的上限取决于你能想到多少边界情况。而在 AI 系统中，边界情况是无限的，因为 AI 的行为本质上是非确定性的。你永远无法通过规则完全约束它。

### 2.2 结果确定性的闭环

结果确定性代表了一种完全不同的思路：与其规范过程，不如定义清晰的目标状态，然后让系统自己找到到达那里的方式。这种方法的关键在于建立一个闭环：执行 → 观察 → 验证 → 纠正。

当我把翻译任务交给 Claude Code 时，这个闭环才真正成为可能。Claude Code 的基本操作单位是文件，文件是有状态的、可持久化的。这意味着 AI 可以看到它之前做了什么，可以运行验证脚本来检查结果，可以根据反馈进行调整。这不是一次性的 API 调用，而是一个迭代的、自纠正的过程。

具体来说，当我定义"翻译完成"的标准时——格式正确、没有残留中文字符、术语一致——我可以把这些标准编码成可执行的检查。AI 不仅要完成翻译，还要运行这些检查，看到失败信息，然后自己修复问题。这个过程可以重复，直到所有检查都通过。关键的转变是：我不再需要预见所有可能的失败模式，我只需要定义什么是成功。

这个原则在成本结构上也有深刻的含义。传统编程中，代码执行几乎免费，但人力成本很高，所以我们投入大量精力来设计完美的逻辑。但在 AI 时代，推理成本在快速下降。多次尝试、检查和纠正的成本往往比为每个长尾失败模式写防御性规则更便宜。这是一个根本性的经济学转变。

### 2.3 验收标准的显式化

结果确定性的前提是能够清晰地定义"完成"。这听起来简单，但实际上是最大的瓶颈。大多数失败不是因为 AI 不够聪明，而是因为 AI 不知道什么叫"完成"。

我用一个比喻来理解这个问题：想象你给一个失忆的实习生分配任务。这个实习生没有任何背景知识，不知道你之前讨论过什么，不知道你的隐含期望。他只能看到你这一次给他的指令。如果你想让他可靠地完成任务，你必须把验收标准写得极其清晰——清晰到他可以根据这些标准判断自己是否完成了。如果他认为还没完成，他知道缺什么。

这正是 Claude Code 的工作方式。当我说"翻译这个文件，然后运行这个 Python 脚本来检查是否有残留中文字符，如果有就修复"时，我实际上是在把隐含的期望变成显式的、可验证的标准。这个转变本身就是最大的杠杆。

### 2.4 架构优于规则

当系统表现不佳时，我们的第一反应往往是"模型不够好"或"需要更多规则"。但实际上，许多失败源于架构设计的不当。Wide Research 的例子很有启发性：与其要求单个 AI 调用完美执行复杂任务，不如将任务分解为多个小步骤，每步都有明确的验收标准。这不是模型魔法，而是管理修复。

同样的原则适用于翻译。当 API 调用在长文本上"摸鱼"时，我曾以为是模型的问题。但实际上，这是一个架构问题：长输出会降低指令遵循能力。解决方案不是要求模型"更聪明"，而是改变架构——让 AI 逐章翻译，每章都有明确的输入和输出，每章都可以独立验证。这样，问题就从"如何让一个 API 调用完美执行"变成了"如何设计一个流程，使得每个小步骤都可靠"。

## 3. 应用判定

### 何时使用

结果确定性适用于任何输出可检查的任务。这包括：

- **格式化任务**：代码生成、文档转换、数据清洗——这些都有明确的成功标准
- **验证任务**：检查是否满足特定条件（没有残留中文、通过测试、符合规范）
- **当护栏规则开始多于产品逻辑时**：这是一个信号，说明你应该转向结果确定性

### 如何实践

1. **先写验收标准**：不要说"生成高质量的代码"，要说"代码必须通过所有单元测试，覆盖率 > 80%，没有安全警告"。把隐含期望变成显式的、可测量的标准。

2. **尽可能把标准固化成可执行检查**：Python 脚本、单元测试、linter、正则表达式——任何能自动验证的东西都应该自动化。这样 AI 就可以自己运行检查，看到失败，然后修复。

3. **让 agent 自己选择方法并迭代**：不要规范 AI 如何做，只定义什么是成功。AI 可能用 regex 检查，也可能用 NLP，也可能用其他方法。只要最终结果满足标准，就是可靠的。

4. **建立反馈循环**：确保 AI 能看到验证的结果，能根据失败信息进行调整。这个闭环是结果确定性的核心。

## 4. 陷阱与边界

### 何时不适用

- **验收标准无法定义**：如果成功本质上是主观的（如"创意写作"），结果确定性会很困难
- **实时性要求极高**：如果验证过程会导致不可接受的延迟，可能需要接受更高的风险
- **成本-收益不匹配**：如果验证成本远高于失败成本，可能不值得投入

### 常见陷阱

1. **虚假的验收标准**：定义了看起来清晰但实际上模糊的标准。例如"代码质量好"不是标准，"通过所有测试且复杂度 < 10"才是。

2. **验证盲目性**：验证规则本身有缺陷，导致通过验证的输出仍然失败。需要定期审查验证规则，确保它们真的在测量你关心的东西。

3. **过度验证**：为低风险任务设计过于复杂的验证流程，导致效率下降。验证应该与风险等级相匹配。

4. **信任漂移**：随着时间推移，逐渐放松验证标准，直到系统变得不可靠。需要定期审查和重新校准。

## 5. 与其他公理的关系

- **A04 可靠性是管理问题**：结果确定性是可靠性管理的核心方法
- **V02 可验证性是信任的地基**：结果确定性依赖于清晰的验收标准和可执行的检查
- **T07 隔离-处理-验证闭环**：结果确定性是这个闭环的验证阶段
- **T01 基础设施优于组件**：结果确定性需要一个支持反馈循环的运行时（如 Claude Code）
